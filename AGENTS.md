## Project Overview

This project modularizes JavaScript ES6 for use in Perchance (https://perchance.org), a procedural generation platform. Modules are served via GitHub + jsDelivr CDN because Perchance does not support bundlers. Each module in `src/modules/` tests a specific plugin or functionality within the Perchance ecosystem.

## Project Structure

- `src/main.js` — Entry point (imports and initializes modules)
- `src/perchance-bridge.js` — Safe bridge to Perchance root API/lists
- `src/constants.js` — Global constants (version, CDN)
- `src/modules/*.js` — Independent modules (renderer, logic, plugin tests)
- `for-perchance.html` — HTML Panel for copy/paste into Perchance
- `for-perchance-list-panel.txt` — List Panel for copy/paste into Perchance
- `test-local.html` — Local testing outside Perchance (open in browser)
- `dist/main.bundle.js` — Single bundle generated by Vite (committed)

## 🪝 Git Pre-Commit Hook: Versioning Automation

This project has a pre-commit hook that automates version updates across multiple files.

### Hook Behavior

The hook runs `scripts/sync-version.cjs` on every commit. This script:

1. **Validates semver format** - Ensures version follows `vMAJOR.MINOR.PATCH` or `MAJOR.MINOR.PATCH`
2. **Scans the entire project** - Finds all version references intelligently
3. **Protects sensitive files** - Never modifies:
   - `CHANGELOG.md` (historical record)
   - `package-lock.json` (auto-generated)
   - `yarn.lock`, `pnpm-lock.yaml` (package manager locks)
4. **Updates only project-specific references** - Avoids false positives on external library versions
5. **Warns about Git status** - Alerts if tags are missing or changes are uncommitted

### What Gets Updated

| File | What is Updated | Pattern |
|------|----------------|---------|
| `package.json` | `"version"` field only | `"version": "X.Y.Z"` |
| `README.md` | Line 1 title only | `# Title (vX.Y.Z)` |
| `for-perchance.html` | CDN URLs to this repo | `cdn.jsdelivr.net/gh/Fahell/test-perchance-git@vX.Y.Z` |
| `src/main.js` | Comments and BASE_URL | `// Version: vX.Y.Z` |
| Other files | Repository-specific CDN URLs | Only URLs containing `test-perchance-git@` |

### What is Protected (Never Updated)

- ❌ External library versions (Three.js 0.160.0, Howler.js 2.23.0, etc.)
- ❌ `CHANGELOG.md` (historical record)
- ❌ `package-lock.json` (auto-generated by npm)
- ❌ Dependency versions in `package.json` (only project version is updated)

### Correct Release Workflow

1. **Update `src/constants.js`** with the new version:
   ```javascript
   export const VERSION = '1.4.1';
   ```

2. **Commit** - the hook will detect the change and update other files automatically:
   ```bash
   git add src/constants.js
   git commit -m "chore: release v1.4.1"
   ```

3. **Add the files updated by the hook** and make the final commit:
   ```bash
   git add .
   git commit --amend --no-edit
   ```

4. **Create the tag and push**:
   ```bash
   git tag -a v1.4.1 -m "Release v1.4.1"
   git push origin main --tags
   ```

### ⚠️ Important

- **Never** create a tag without first updating `constants.js`
- The hook **always runs** on every commit (idempotent, ~50ms)
- If everything is already synchronized, the hook makes no changes
- The hook warns if the tag doesn't exist or hasn't been pushed
- Always verify that `constants.js`, `package.json`, and `for-perchance.html` have the same version before creating a tag

### Common Error Example

❌ **Wrong:**
```bash
# Updates only package.json manually
npm version 1.4.0
git tag -a v1.4.0 -m "Release v1.4.0"
# constants.js is still at 1.3.0! The hook was not triggered.
```

✅ **Correct:**
```bash
# Update constants.js first
sed -i "s/VERSION = '1.3.0'/VERSION = '1.4.0'/" src/constants.js
git add src/constants.js
git commit -m "chore: release v1.4.0"
# Hook updates package.json and for-perchance.html automatically
git add .
git commit --amend --no-edit
git tag -a v1.4.0 -m "Release v1.4.0"
git push origin main --tags
```


### Search Strategy

The script uses **recursive search** with intelligent patterns to find version references:

1. **Recursive file scanning** - Scans all text files in the project
2. **Protected directories** - Excludes: `node_modules/`, `dist/`, `.git/`
3. **Protected files** - Never modifies: `CHANGELOG.md`, `package-lock.json`, lock files
4. **Pattern matching** - Only updates:
   - Repository-specific CDN URLs (containing `test-perchance-git@`)
   - Comments with explicit version markers (`// Version: vX.Y.Z`)
   - Project constants (`BASE_URL`)
   - HTML comments with version

**Why recursive search?**
- ✅ Automatic - no need to maintain a list of files
- ✅ Scalable - works with any project structure
- ✅ Defensive - patterns prevent false positives
- ✅ Maintenance-free - new files are automatically included

**To add new file types or patterns:**
Edit `scripts/sync-version.cjs` and add patterns to the `patterns` array.

### Best Practices for Version References

**Use template literals with VERSION constant** for logs and dynamic strings:

```javascript
// ✅ Correct - automatically stays in sync
import { VERSION } from './constants.js';
console.log(`🚀 [Main] Iniciando jogo (Vite bundle ${VERSION})`);

// ❌ Wrong - requires manual update or script pattern
console.log('🚀 [Main] Iniciando jogo (Vite bundle v1.3.0)');
```

This approach:
- Eliminates hardcoded versions in source code
- Reduces dependency on sync script for logs
- Ensures logs always reflect the current version
- Makes code more maintainable

### Manual Sync (If Needed)

You can run the sync script manually to verify or force synchronization:

```bash
node scripts/sync-version.cjs
```

This will:
- Show the current version in `constants.js`
- List all files that would be updated
- Show Git status warnings (missing tags, uncommitted changes)

## Defensive Restrictions

- Never execute destructive commands without explicit confirmation
- Never modify files in `/mnt/c/` except editor configs when requested
- Perchance plugins require specific DOM context; `null.dataset` errors are known limitations (see README)
- jsDelivr caches by tag; never trust versionless URLs (`@main`) for production
- Validate `git status` before any merge/rebase operation

## Project Scripts

All scripts are in `scripts/` and should be executed directly in WSL2:

| Script | Function | Usage |
|---|---|---|
| `scripts/release.js` | Automated release (updates versions, builds, commits, tags) | `npm run release 1.4.1` |
| `scripts/sync-version.cjs` | Synchronize version across project files | `node scripts/sync-version.cjs` |
| `scripts/dev-server.sh` | Local HTTP server for testing | `./scripts/dev-server.sh [port]` |
| `scripts/sync.sh` | Sync with remote repository | `./scripts/sync.sh` |
| `scripts/setup.sh` | Configure hooks and verify tools | `./scripts/setup.sh` |
| `scripts/check-cdn.sh` | Check jsDelivr CDN availability | `./scripts/check-cdn.sh vX.Y.Z` |

> ⚠️ **Note:** The `.bat` wrappers were removed. Execute `.sh` scripts directly.
> The `dev-server.sh` uses `python` (pyenv shim), not `python3`.

## Build Configuration

### Vite Configuration

The project uses Vite to bundle all modules into a single file. Key configuration:

- **Entry point:** `src/main.js`
- **Output:** `dist/main.bundle.js` (196KB, 39KB gzipped)
